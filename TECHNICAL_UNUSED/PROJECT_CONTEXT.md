# PROJECT_CONTEXT.md

## 1. Что такое ОТ-Монитор

ОТ-Монитор — демонстрационный локальный MVP системы информирования инженера по охране труда о нормативных изменениях.

Главная функция продукта — не просто карточки документов, а регулярный мониторинг нормативных источников.

Система должна помогать инженеру:

* хранить список источников мониторинга;
* запускать проверку источников вручную кнопкой "Проверить сейчас";
* в будущем запускать проверку автоматически по расписанию;
* находить новые или потенциально важные документы;
* сохранять ссылку на первоисточник;
* показывать путь к сохранённой копии документа;
* кратко объяснять суть найденного документа;
* показывать, какие типы внутренних документов компании могут потребовать проверки;
* дать инженеру поставить статус обработки;
* дать инженеру написать комментарий;
* сохранить решение инженера.

Система не заменяет инженера, не делает юридическое заключение и не изменяет внутренние документы автоматически.

## 2. Главная логика продукта

Основной рабочий сценарий:

1. Система хранит список источников мониторинга.
2. Система хранит настройки расписания проверки.
3. Проверка может запускаться:

   * вручную кнопкой "Проверить сейчас";
   * автоматически по расписанию: ежедневно, еженедельно или ежемесячно.
4. Система проверяет источники.
5. Система находит новый или потенциально важный документ.
6. Система сохраняет ссылку на первоисточник.
7. Система сохраняет копию документа или путь к файлу.
8. Система добавляет документ в ленту найденных документов.
9. Система показывает карточку документа.
10. Система связывает документ с типами внутренних документов компании.
11. Инженер принимает решение.
12. Решение инженера сохраняется.

## 3. Важное ограничение текущего локального MVP

В текущей первой версии реальный парсинг сайтов пока не делается.

Допускается имитация проверки источников на тестовых данных.

Но интерфейс и структура проекта должны показывать будущую логику:

источники мониторинга → расписание проверки → журнал проверок → найденные документы → карточка документа → решение инженера.

То есть кнопку "Проверить сейчас" нельзя воспринимать как просто декоративную кнопку. Это задел под будущий модуль мониторинга источников.

## 4. Ограничения текущей технической реализации

Текущая версия должна оставаться максимально простой:

* только HTML, CSS, JavaScript;
* данные пока могут быть тестовыми;
* сохранение решений пока через localStorage;
* запуск простым открытием index.html в браузере;
* без сервера;
* без базы данных;
* без авторизации;
* без React;
* без Node.js;
* без npm;
* без Vite;
* без package.json.

## 5. Что должно быть в интерфейсе MVP

В интерфейсе должны быть или постепенно появиться:

* главная страница;
* кнопка "Проверить сейчас";
* карточки показателей;
* список найденных документов;
* карточка выбранного документа;
* список источников мониторинга;
* расписание проверок;
* журнал проверок;
* типы внутренних документов;
* блок решения инженера;
* статус обработки;
* комментарий инженера;
* блок будущего ИИ/RAG;
* блок будущего черновика рекомендации.

## 6. Источники мониторинга

В будущем система может проверять следующие источники:

* Официальный портал правовой информации;
* Минтруд России;
* АКОТ / ЕИСОТ Минтруда;
* МЧС России;
* Роспотребнадзор;
* Роструд / Онлайнинспекция;
* Минздрав России;
* Росстандарт;
* Правительство РФ;
* Национальный антитеррористический комитет;
* regulation.gov.ru — только как источник проектов нормативных актов.

Для первого MVP можно использовать 3–5 источников и тестовые данные.

## 7. Расписание проверок

В MVP должна быть предусмотрена логика расписания проверок.

Возможные варианты:

* ежедневно;
* раз в неделю;
* раз в месяц;
* только вручную.

Даже если автоматический запуск пока имитируется, в данных и интерфейсе должна быть видна будущая настройка расписания.

## 8. Журнал проверок

После проверки источников система должна фиксировать результат в журнале проверок.

В журнале проверок должны быть поля:

* дата и время проверки;
* источник;
* результат проверки;
* сколько документов найдено;
* сколько новых документов добавлено;
* ошибка, если источник недоступен.

В текущем локальном MVP журнал может быть тестовым или имитированным.

## 9. Найденный документ

Карточка найденного документа должна содержать:

* название документа;
* источник;
* ссылку на первоисточник;
* сохранённый файл или путь к файлу;
* дату обнаружения;
* раздел;
* краткую суть;
* типы внутренних документов для проверки;
* статус обработки;
* комментарий инженера;
* блок будущего ИИ/RAG;
* блок будущего черновика рекомендации.

## 10. Разделы документов

Используются разделы:

* Охрана труда;
* Медицина;
* Пожарная безопасность;
* ГО и ЧС;
* Антитеррор.

## 11. Статусы обработки

Используются статусы:

* Новое;
* На проверке;
* Принято в работу;
* Не относится;
* Требует консультации;
* Закрыто.

## 12. Типы внутренних документов

Система работает не с конкретным магазином, а с типом внутреннего документа.

Примеры типов:

* Журнал инструктажа по охране труда;
* Журнал учета посещаемости и проведения занятий по охране труда;
* Журнал учета инструкций по охране труда;
* Журнал учета выдачи инструкций по охране труда;
* Журнал регистрации несчастных случаев;
* Журнал регистрации микротравм;
* Инструкции по охране труда;
* Положения и нормы выдачи по охране труда;
* Карточки СИЗ и смывающие средства;
* Протоколы по охране труда;
* Отчет о проведении профессиональных рисков;
* План по улучшению мероприятий по охране труда;
* Журнал учета выдачи направлений;
* Таблица медкомиссий и санитарного минимума;
* Журнал инструктажа по пожарной безопасности;
* Журнал ЭСПЗ / поверки огнетушителей;
* Паспорта огнетушителей;
* Инструкции по пожарной безопасности;
* План по пожарной безопасности;
* Журнал регистрации инструктажа по ГО и ЧС;
* Журнал учета посещаемости и проведения занятий по ГО и ЧС;
* Журнал учета инструктажей по антитеррористической безопасности;
* Инструкции по ГО и ЧС, антитеррористической безопасности;
* Функциональные обязанности;
* Планы по ГО и ЧС, антитеррористической безопасности;
* Акты проведения тренировок по покиданию и укрытию.

## 13. Будущий ИИ/RAG

MVP должен быть подготовлен к будущей ИИ/RAG-архитектуре.

В будущем система сможет:

* хранить очищенный текст найденного документа;
* делать краткое объяснение;
* выделять возможные требования;
* связывать внешний документ с типами внутренних документов;
* учитывать решение инженера;
* готовить черновик рекомендации.

Важно: черновик рекомендации не должен применяться автоматически.

Обязательная пометка:
"Черновик, требует проверки инженером."

## 14. Что нельзя делать без отдельного указания

Codex не должен самовольно:

* переводить проект на React;
* создавать сервер;
* добавлять Node.js;
* добавлять npm-зависимости;
* создавать package.json;
* подключать базу данных;
* делать авторизацию;
* делать реальный парсинг сайтов;
* обходить капчи;
* работать с личными кабинетами;
* усложнять архитектуру;
* удалять существующую рабочую логику;
* менять способ запуска через index.html.

## 15. Как работать дальше

Перед каждой доработкой нужно:

1. Прочитать PROJECT_CONTEXT.md.
2. Понять текущую структуру проекта.
3. Менять только те файлы, которые нужны для конкретной задачи.
4. Делать маленькие понятные изменения.
5. Сохранять возможность запуска через index.html.
6. Не добавлять сложные технологии без отдельной команды пользователя.

Главный принцип проекта:

сначала простой локальный MVP → потом имитация мониторинга → потом структура источников и расписаний → потом реальный мониторинг → потом ИИ/RAG.
